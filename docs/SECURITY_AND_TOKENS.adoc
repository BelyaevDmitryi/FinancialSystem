= Безопасность и токены (продакшен)

== Архитектура аутентификации

* *Единая точка входа:* все запросы к API проходят через ApiGateway. Только Gateway проверяет JWT и проставляет заголовки `X-User-Id` и `X-User-Roles` для микросервисов.
* *Выдача токенов:* UserService выдаёт access- и refresh-токены при логине и при обмене refresh на новую пару.
* *Микросервисы* не проверяют JWT — они доверяют заголовкам `X-User-Id` и `X-User-Roles`, установленным Gateway. Данные изолированы по пользователю: каждый сервис фильтрует по `userId` из заголовка.

=== Типы токенов

|Токен|Назначение|Срок жизни (по умолчанию)|Где хранится (фронт)|
|Access|Авторизация запросов (Bearer)|15 мин|localStorage|
|Refresh|Обмен на новую пару access + refresh без пароля|7 дней|localStorage|

* В продакшене для уменьшения риска XSS можно хранить токены в httpOnly cookie (требует доработки бэкенда и фронта).

=== Поток при логине

1. Фронт отправляет POST `/api/auth/signin` (логин/пароль).
2. UserService проверяет учётные данные, возвращает access + refresh токены и данные пользователя.
3. Фронт сохраняет токены и при каждом запросе добавляет заголовок `Authorization: Bearer <access>`.
4. ApiGateway проверяет access-токен, извлекает `userId` и роли, добавляет заголовки `X-User-Id` и `X-User-Roles` и передаёт запрос в микросервис.

=== Поток при истечении access-токена (401)

1. Фронт получает 401 на любой запрос.
2. Фронт вызывает POST `/api/auth/refresh` с телом `{ "refreshToken": "..." }` (без Authorization).
3. UserService проверяет refresh-токен и возвращает новую пару access + refresh.
4. Фронт сохраняет новые токены и повторяет исходный запрос.
5. Если refresh невалиден или истёк — фронт очищает хранилище и перенаправляет на страницу входа.

== Мультипользовательское использование

* *Один браузер/вкладка = один активный пользователь.* В одном сеансе браузера хранится один набор токенов (один пользователь).
* *Разные пользователи* — это разные устройства или браузеры (или разные профили браузера). У каждого свой логин и свой набор токенов.
* *Изоляция данных:* ордера, счета брокера, боты, профиль привязаны к `userId` из токена. Пользователь A не видит и не может изменить данные пользователя B.
* *Админ-функции* (например, глобальная статистика) доступны только при наличии роли `ROLE_ADMIN` или `ROLE_OWNER` в токене; Gateway передаёт роли в заголовке `X-User-Roles`.

== Конфигурация для продакшена

* *Обязательно задать через переменные окружения:*
** `JWT_SECRET` — один и тот же для UserService и ApiGateway (длинная случайная строка, минимум 256 бит для HS256).
** В UserService (опционально): `JWT_EXPIRATION_MS` (access, по умолчанию 15 мин), `JWT_REFRESH_EXPIRATION_MS` (refresh, по умолчанию 7 дней).
* *Не задавать в продакшене:* `BROKER_ACCOUNT_ID` для TradingTerminalService — счёт брокера берётся из UserService по текущему пользователю (свой счёт у каждого пользователя).

== Чек-лист перед выкладкой

* [ ] Установлен уникальный `JWT_SECRET` в окружении UserService и ApiGateway.
* [ ] Публичные пути в Gateway ограничены: `/api/auth/signin`, `/api/auth/signup`, `/api/auth/refresh`, документация, actuator.
* [ ] CORS и допустимые origin настроены под домен фронта (не `*` в продакшене, если нужна строгая политика).
* [ ] Токены не логируются и не попадают в метрики/трассировки.
