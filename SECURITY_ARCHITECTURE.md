# Архитектура безопасности системы

## Анализ текущей ситуации

### Проблемы, которые были обнаружены:

1. **JWT валидация только в UserService**
   - JWT токены валидировались только для запросов к UserService
   - Другие микросервисы не были защищены
   - Любой мог отправить запрос с поддельным заголовком `X-User-Id`

2. **ApiGateway не валидировал токены**
   - Gateway просто проксировал запросы без проверки
   - Отсутствовала централизованная точка аутентификации

3. **Дублирование кода**
   - Если бы JWT был реализован в нескольких сервисах, пришлось бы дублировать код
   - Изменение секрета потребовало бы изменений во всех сервисах

4. **ACL не реализован**
   - Конфигурация ACL существует, но не используется
   - Spring Security ACL требует JDBC, а проект использует MongoDB
   - Нет аннотаций `@PreAuthorize` с `hasPermission`

## Реализованное решение

### Централизованная аутентификация в ApiGateway

**Преимущества:**
- ✅ Единая точка валидации JWT токенов
- ✅ Все микросервисы автоматически защищены
- ✅ Упрощение поддержки (один секрет, одна конфигурация)
- ✅ Автоматическая передача `X-User-Id` в микросервисы после валидации

**Что было сделано:**

1. **ApiGateway:**
   - Добавлены JWT зависимости
   - Создан `JwtUtils` для валидации токенов
   - Создан `JwtAuthenticationFilter` (GlobalFilter) для валидации всех запросов
   - Публичные пути (`/api/auth/**`, `/api-docs`, `/actuator`) пропускаются без валидации
   - После валидации токена `username` передается в заголовке `X-User-Id` в микросервисы

2. **UserService:**
   - Убрана JWT валидация (оставлена только генерация токенов)
   - Упрощена конфигурация безопасности
   - Сервис теперь доступен для внутренних вызовов без аутентификации

## Рекомендации по ACL

### Варианты реализации ACL:

#### Вариант 1: Отдельный AuthService (рекомендуется)
Создать отдельный микросервис `AuthService` для:
- Генерации и валидации JWT токенов
- Управления ACL (разрешениями доступа)
- Хранения ролей и прав пользователей

**Преимущества:**
- Разделение ответственности (Single Responsibility Principle)
- Централизованное управление безопасностью
- Легче масштабировать и поддерживать

#### Вариант 2: ACL в ApiGateway
Реализовать проверку разрешений в ApiGateway после валидации JWT.

**Преимущества:**
- Все проверки в одном месте
- Микросервисы не знают о правах доступа

**Недостатки:**
- ApiGateway становится более сложным
- Нужно хранить информацию о разрешениях

#### Вариант 3: ACL в каждом сервисе
Каждый сервис проверяет свои разрешения.

**Недостатки:**
- Дублирование логики
- Сложность поддержки

### Рекомендация: Вариант 1 (AuthService)

**Структура AuthService:**
```
AuthService/
├── JWT генерация и валидация
├── ACL управление
├── Роли и права пользователей
└── Интеграция с UserService для получения данных пользователя
```

**Интеграция:**
- ApiGateway валидирует JWT через AuthService
- ApiGateway проверяет разрешения через AuthService
- Микросервисы получают только `X-User-Id` и не знают о правах доступа

## Текущая архитектура

```
Client → ApiGateway (JWT валидация) → Микросервисы (X-User-Id)
                ↓
         UserService (генерация JWT)
```

## Важно

- **НЕ размещайте JWT валидацию в нескольких сервисах** - это приведет к дублированию кода и сложности поддержки
- **Централизуйте аутентификацию** в ApiGateway или отдельном AuthService
- **ACL лучше вынести** в отдельный сервис для лучшей архитектуры
