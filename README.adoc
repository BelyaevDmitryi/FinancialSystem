== Идея

Микросервисная платформа для анализа портфеля, торговых роботов и стратегий на российском рынке (Т-Инвестиции и др.). Пользователь проходит аутентификацию через API Gateway, управляет портфелем и ботами, получает котировки, индикаторы, журнал сделок; для офлайн-анализа доступны исторические свечи, бэктест и grid-оптимизация параметров.

== Стек и инфраструктура

* **Backend:** Java 17, Spring Boot 3, Spring Cloud (Eureka, Gateway), OpenFeign, Spring Security + JWT
* **Хранение:** **PostgreSQL** (отдельные БД/схемы по сервисам), **Redis** (кэш цен в PriceService)
* **Сборка:** Maven (обёртки `mvnw` в репозитории нет)
* **Контейнеризация:** Docker Compose (`docker-compose.yml`, тестовые профили — см. link:DEVELOPER.md[DEVELOPER.md])
* **Миграции БД:** Flyway (где подключено в модуле)
* **API:** REST через Gateway `:8090`, OpenAPI (springdoc) по сервисам

== Архитектура сервисов

Все модули перечислены в корневом `pom.xml`. Внешний клиент обращается только к **API Gateway**; сервисы регистрируются в **Eureka Server**.

| Сервис | Назначение |
|--------|------------|
| ApiGateway | Единая точка входа, маршрутизация `/api/**`, JWT |
| EurekaServer | Service discovery |
| UserService | Пользователи, аутентификация, портфель |
| PriceService | Актуальные цены, Redis-кэш |
| BrokerIntegrationService | Адаптеры брокера (TINKOFF, MOCK в тесте) |
| AnalyticsService | SMA, EMA, MACD, волатильность |
| TradingTerminalService | Ордера, исполнение через брокера |
| TradingBotService | Торговые роботы, scheduler стратегий |
| JournalService | Журнал сделок и позиций |
| MarketHistoryService | Хранение и импорт OHLCV-свечей |
| BotOptimizationService | Бэктест (`/api/backtest/**`) и grid-оптимизация |
| AdminPanelService | Админ-статистика |
| DashboardService | Данные дашборда |

Диаграммы: link:docs/architecture/c4-containers-financialsystem.puml[C4 контейнеры], link:docs/architecture/c4-comparison-osengine-vs-financialsystem.puml[сравнение с OsEngine].

== Сценарий использования

. Пользователь регистрируется / входит через Gateway (`POST /api/auth/signin`), получает JWT.
. Запрашивает портфель и котировки: UserService → PriceService (кэш Redis) → при необходимости BrokerIntegrationService → API брокера.
. Создаёт ордер через TradingTerminalService (`POST /api/orders`); исполнение идёт через брокерский адаптер; JournalService фиксирует сделки.
. Создаёт торгового бота (`POST /api/bots`): TradingBotService по расписанию запрашивает AnalyticsService и PriceService, при сигнале создаёт ордер в терминале.
. Для офлайн-анализа: импорт свечей в MarketHistoryService, прогон бэктеста и оптимизации параметров через BotOptimizationService.

Тестовый стек без реального Tinkoff API, health-check и curl-примеры — в link:DEVELOPER.md[DEVELOPER.md].
